# v3 Design — "Catan-Lite" GNN Training Target

**Status:** Approved (2026-05-02)

**Predecessor:** v2 — see `docs/superpowers/specs/2026-04-30-v2-restart-full-game-design.md`

## 1. Goal

**Train a stronger GNN faster** by simplifying the Catan win condition. v2's full-rule training did not produce a GNN that beats `LookaheadMcts` (0/4 seats in the Phase 3 4-way tournament). v3 hypothesizes that a shorter game with a cleaner reward signal will let the GNN+MCTS loop converge to "GNN beats lookahead" on our hardware, after which the architecture can be ported back to full Catan.

The simplifications are scaffolding — they must be reversible (engine flags, not hardcoded constants) so the same GNN code path runs against full Catan once the simpler problem is solved.

## 2. Non-Goals

- **Not** a permanent rule fork. v3 is a training/diagnostic vehicle, not a new game.
- **Not** a v2 engine rewrite. The v2 engine, observation, action space, GNN architecture, recorder schema, and training pipeline are reused unchanged.
- **Not** removing dev cards or trades. Only the win condition simplifies; everything else stays full-Catan.
- **Not** introducing a `RuleSet` trait or `engine_v3.rs` fork. See §10 ("Approaches considered, rejected").

## 3. Rule Changes

Two changes, both runtime-flag-controlled with v2 defaults:

| Change | v2 default | v3 setting | Mechanism |
|---|---|---|---|
| Win-condition VP threshold | 10 | **5** | `vp_target: u8` arg to `Engine::new` |
| Bonus VPs (longest road, largest army) | +2 each, transferable | **disabled (no +2 awarded)** | `bonuses: bool` arg to `Engine::new` |

**Bonus-disabled semantics:** the engine still tracks `longest_road_length`, `longest_road_holder`, `largest_army_holder` and still emits them in `observation_for()`. They simply don't contribute to `state.vp[p]`. This keeps the GNN observation identical between v2 and v3 — only the labels (winner determination, terminal value) differ.

**Everything else unchanged from v2:** dev cards, player-to-player trades, maritime trades (4:1/3:1/2:1), ports, robber/discard, ABC balanced map, 280-action space, F_VERT=13, N_SCALARS=59.

## 4. Engine API

```rust
impl Engine {
    pub fn new(seed: u64) -> Self {
        Self::with_rules(seed, 10, true)  // v2 default preserved
    }

    pub fn with_rules(seed: u64, vp_target: u8, bonuses: bool) -> Self {
        // ...
    }
}
```

**PyO3 surface:** add `Engine.with_rules(seed, vp_target, bonuses)` classmethod. `Engine(seed)` keeps current behavior.

**State:** add `vp_target: u8` and `bonuses_enabled: bool` to `GameState`. Both are immutable for the lifetime of the engine.

**Constants:** `WIN_VP` becomes the v2 *default*, not a global. All `>= WIN_VP` checks use `state.vp_target` instead.

**Bonus award sites** (in `rules.rs`):
- `update_longest_road`: skip the `state.vp[holder] += 2` line when `!state.bonuses_enabled`. Still update `longest_road_length` and `longest_road_holder`.
- `update_largest_army`: same pattern.

**Determinism:** `vp_target` and `bonuses` participate in the seed-stable hash so v3 and v2 runs of the same seed are deterministically distinguishable but neither is broken.

## 5. Data Generator: Lookahead-MCTS with VP-Aware Sim Schedule

All four seats are `LookaheadMcts` players. Per move, the simulation budget is set as a function of the acting player's current VP at the root:

```python
def sims_for_player(acting_vp: int) -> int:
    return max(50, round(200 * 0.7 ** acting_vp))
```

| acting_vp | 0 | 1 | 2 | 3 | 4 |
|---|---|---|---|---|---|
| sims | 200 | 140 | 98 | 69 | 50 |

**Why exponential and floored:** spend the most search far from the win where positions are most diverse and lookahead pays off; tail off as the game closes; floor at 50 so endgame moves still get a meaningful policy signal for the GNN to learn.

**Why per-player VP (not max VP):** the player to move has the most actionable choices when they're far from winning. Tying the budget to *their* VP reflects how much they have left to decide, not how close the *game* is to ending.

**No separate early-exit logic:** with engine `vp_target=5`, the search naturally terminates at 5 VP. The "early-exit on VP=5" idea from earlier discussion collapses into the engine flag.

## 6. Data Pipeline

**Recorder schema:** unchanged. v2's `games.parquet` (with `action_history`, `winner`, `seed`, `schema_version=2`) and `moves.parquet` (with `mcts_visit_counts`, `legal_action_mask`, `current_player`, `move_index`) work as-is. The recorder doesn't know about `vp_target`/`bonuses`; the values land in the action history because the engine state captures them implicitly through the win condition.

**Action history portability:** an action sequence recorded under v3 rules can be replayed under v2 rules. It will diverge whenever the v3 game ended at 5 VP but v2 wouldn't have. The `CatanReplayDataset` only replays up to a recorded `move_index`, so this divergence is fine — what matters is reaching the position, not playing past it.

**New artifact:** `runs/v3/<timestamp>_e9_v3_data/` parquet shards from the v3 sweep. Naming convention follows v2 sweep dirs.

## 7. Training Pipeline

**Reused without changes:**
- `state_to_pyg.py` (observation → HeteroData)
- `gnn_model.py` (80k-param SAGEConv ×2)
- `train.py` (CLI flags, value+policy heads, val split, checkpointing)
- `dataset.py` (`CatanReplayDataset`, `CachedDataset`, `RotatedDataset`)
- `rotation.py` (6-fold hex symmetry augmentation)

**Why no GNN/architecture changes:** the goal is to test the *training-data hypothesis*. Changing the model at the same time muddies attribution. If v3 training produces a GNN that beats lookahead, we've validated the architecture; if not, we've isolated the data signal.

**Targets initially:**
- 1 small validation run (≤1k games, 5 epochs) to prove the pipeline end-to-end
- Volume target for the production run: deferred until §11 step 1 (game-length measurement) lands

## 8. Tournament / Evaluation

**`e9_v3_tournament.py`** — copy of `e7_gnn_tournament.py` parameterized to construct engines with `(vp_target=5, bonuses=False)`. Four seats: `PureGnn`, `LookaheadMcts`, `Random`, `GnnMcts`. Same N-seed grid, same per-seat win counting.

**Cross-rule sanity check** — load the v3-trained checkpoint and run a smaller tournament under v2 rules `(vp_target=10, bonuses=True)` to verify features transfer.

## 10. Approaches Considered, Rejected

**Approach B — Forked engine (`engine_v3.rs`):** maximum freedom but doubles the test surface, contradicts the "port back to full Catan" path, and the goal is faster training, not engine experimentation. Rejected.

**Approach C — `RuleSet` trait + dynamic dispatch:** premature abstraction (we have *two* rule sets, not N), introduces vtable overhead in the engine hot loop. Rejected.

**Approach A — Two flags on the constructor (chosen):** smallest diff, one engine binary serves both rule sets, v2 tests parameterize over both modes, no Python adapter changes.

## 11. Implementation Order

1. **Measure v3 game length** — minimal patch (constants + flags), then run pure-random and current-`LookaheadMcts` baselines under v3 rules. Establishes time-per-game; calibrates volume target for the production sweep.
2. **Engine flag plumbing + tests** — add `vp_target` and `bonuses_enabled` to `GameState`, route through `Engine::with_rules`, gate the `+2` lines, parameterize property tests over both modes.
3. **PyO3 + Python adapter** — expose `Engine.with_rules` to Python, thread the flags through `catan_mcts.adapter`.
4. **Data-gen player** — `LookaheadMctsV3` with the exponential VP-aware sim schedule. Keep the v2 `LookaheadMcts` available for tournament comparison.
5. **Validation sweep** — 1k games, train 5 epochs, run e9 tournament, write a journal entry. Decide on production scale.
6. **Production sweep + training** — sized from step 1.
7. **Cross-rule sanity check** — v3-trained GNN on v2 engine.

## 12. Success Criteria

1. Engine compiles. `cargo test` passes when `vp_target=10, bonuses=true` (v2 invariants preserved). New tests pass for the v3 mode.
2. Lookahead-MCTS data generator produces ≥1k v3 games in ≤15 min on the dev machine.
3. After ≥5 epochs of training on a v3 dataset, a 4-way `e9` tournament gives `PureGnn` or `GnnMcts` ≥1/4 seat wins (currently 0/4 in v2).
4. The trained model loaded under v2 rules wins more than `Random` in a small smoke tournament — features transfer.

## 13. Open Questions (Deferred)

- Production data-generation scale — set after §11 step 1.
- Whether to also enable rotation augmentation (`RotatedDataset`) on the v3 training run by default. Tentatively yes, since it's free, but confirm during §11 step 5.
- Whether to add a v3-mode flag to the playback viewer for visual inspection of v3 games. Probably yes, low priority.
- Whether `LookaheadMctsV3` should also use the v2 `length-discount` rollout (currently DECAY=0.999). Default: yes — keep it on, since it's orthogonal to the VP changes.
